# 高校校园助手 APP 需求说明书

## 一、文档概述

### 1.1 文档目的

本需求说明书明确高校校园助手 APP（基于 HarmonyOS NEXT 开发）的产品定位、功能需求、技术要求、界面设计规范及开发约束等内容，为 2-3 人学生开发团队提供清晰的开发指引，确保项目开发符合《移动开发技术》课程教学大纲要求，且能在 2 个月左右完成开发实现。

### 1.2 适用范围

本说明书适用于参与该 APP 开发的全体学生成员、课程指导教师，作为项目开发、测试、验收的核心依据。

### 1.3 开发背景

结合《移动开发技术》课程 HarmonyOS NEXT 开发教学内容，面向高校学生校园生活场景，开发一款轻量化、实用性强的校园助手 APP，整合校园生活、学习、社交等核心需求，同时让开发团队熟练运用 HarmonyOS NEXT 相关开发技术，完成课程期末考核实践项目。

### 1.4 开发约束

- **技术栈**：严格基于 HarmonyOS NEXT、ArkTS 语言、ArkUI 开发框架，涵盖课程大纲中 DevEco Studio 使用、布局设计、组件开发、页面路由、数据管理、多媒体开发等核心内容；
- **团队与周期**：2-3 人初学者团队，开发周期约 2 个月（含需求分析、开发、测试、优化）；
- **功能规模**：轻量化核心功能，避免过度复杂开发，注重功能完整性和技术点覆盖，符合课程教学要求。

## 二、项目整体要求

### 2.1 产品定位

面向高校全体学生，基于 HarmonyOS NEXT 的移动端校园助手 APP，聚焦校园信息查询、学习辅助、生活服务三大核心场景，提供简洁、高效、贴合校园实际的功能服务，满足学生日常校园生活的基础需求，同时适配移动端独特的操作习惯和使用场景。

### 2.2 技术要求

- **开发工具**：使用 DevEco Studio 最新版，适配 HarmonyOS NEXT Api 12；
- **开发语言**：ArkTS（贴合课程大纲中 ArkTs 基础知识教学内容）；
- **开发框架**：ArkUI，熟练运用线性布局（Row/Column）、相对布局（RelativeContainer）等布局方式；
- **技术点覆盖**：需涵盖课程大纲中的组件开发、自定义组件、页面路由与导航、Stage 模型窗口管理、本地数据存储、多媒体开发、事件绑定与组件交互等核心授课内容，侧重移动端的屏幕适配、沉浸式体验、本地文件读取等独特场景开发；
- **项目架构**：采用 HarmonyOS NEXT 的 Stage 模型，实现应用窗口管理、生命周期管理；
- **兼容性**：适配 HarmonyOS NEXT 主流移动端设备，保证界面自适应、功能正常运行。

### 2.3 团队分工建议（2-3 人）

结合初学者能力和开发周期，建议分工如下，确保每人参与核心技术点开发：

- 1 名开发人员：负责基础框架搭建、页面布局设计、基础组件开发（按钮、单选框、图片、进度条等）、Stage 模型窗口管理；
- 1 名开发人员：负责自定义组件开发、页面路由与导航、事件绑定与组件交互、校园信息模块开发；
- 可选 1 名开发人员：负责本地数据存储、多媒体开发（图片 / 音频）、生活服务模块开发、测试与优化（2 人团队则由前两人分摊该部分工作）。

## 三、功能需求

本 APP 功能分为首页、校园信息、学习助手、生活服务、我的五大模块，各模块功能围绕校园实际需求设计，同时严格覆盖课程教学大纲的核心技术点，所有功能均基于 HarmonyOS NEXT 移动端特性开发。

### 3.1 通用基础功能

#### 3.1.1 应用启动与窗口管理

- **启动页**：展示 APP 名称、logo，实现沉浸式启动界面（贴合 Stage 模型窗口沉浸式能力教学内容）；
- **主窗口设置**：设置应用主窗口属性，适配移动端不同屏幕尺寸，支持窗口大小自适应；
- **悬浮窗功能**：提供校园待办悬浮窗，可在 APP 后台时显示核心待办信息（贴合 Stage 模型悬浮窗开发教学内容）；
- **生命周期管理**：实现 Stage 模型下应用、页面的生命周期管理，保证应用前后台切换、页面跳转的稳定性。

#### 3.1.2 页面导航与路由

- **底部 Tab 导航**：实现首页、校园信息、学习助手、生活服务、我的五大模块的底部 Tab 切换，支持自定义 Tab 样式（贴合导航和页面路由教学内容）；
- **页面跳转**：实现各模块内子页面的路由跳转，支持自定义转场动画和过渡效果；
- **返回与刷新**：支持移动端常用的返回上一页、下拉刷新功能，符合移动端操作习惯。

#### 3.1.3 通用组件交互

- **基础组件**：使用 Button、Radio、Toggle、Progress、Image 等 ArkUI 基础组件，实现按钮点击、单选选择、开关切换、进度展示、图片加载等功能（贴合组件教学内容）；
- **事件绑定**：为所有可交互组件绑定点击、滑动、长按等事件，实现组件与用户的交互响应（贴合组件交互与事件绑定教学内容）；
- **列表组件**：使用 ArkUI 列表组件展示校园资讯、课程表、失物招领等列表类信息，支持上拉加载更多。

### 3.2 首页模块

- **功能入口**：以九宫格 / 卡片形式展示校园信息、课程表、成绩查询、失物招领、校园地图等核心功能的快捷入口，支持自定义入口排序；
- **校园公告**：展示校园最新公告（如活动通知、考试安排、放假通知），支持公告标题、时间、简要内容展示，点击进入详情页；
- **天气查询**：展示当前校园所在地区的天气情况（温度、天气状况、风力），基于静态数据实现（简化开发，侧重界面展示）；
- **待办提醒**：展示用户添加的学习、生活待办事项，支持待办事项的新增、删除、标记完成。

> **技术点覆盖**：ArkUI 布局设计（线性 / 相对布局）、基础组件使用、事件绑定、列表组件展示、自定义卡片组件。

### 3.3 校园信息模块

#### 3.3.1 校园公告详情

- 展示公告完整内容，支持图片、文字混排，支持公告收藏、分享（分享功能简化为本地复制链接）；
- 支持公告分类筛选（如学校通知、学院通知、社团通知）。

#### 3.3.2 校园资讯

- 展示校园新闻、社团活动、讲座信息等资讯内容，支持资讯分类、搜索；
- 资讯详情页支持图片放大查看、文字复制。

#### 3.3.3 校园地图

- 提供校园平面地图，标注教学楼、图书馆、食堂、宿舍、操场等核心建筑；
- 支持建筑名称搜索，点击建筑标注显示建筑介绍、位置信息。

> **技术点覆盖**：页面路由与转场、图片组件开发、自定义筛选组件、搜索功能实现、本地静态数据管理。

### 3.4 学习助手模块（核心模块）

#### 3.4.1 课程表管理

- **课程表添加**：支持用户手动添加课程信息（课程名称、授课老师、上课时间、上课地点、学分），支持单选框选择上课周次；
- **课程表展示**：以周视图（周一至周日）形式展示课程表，适配移动端竖屏展示，支持滑动切换周次；
- **课程提醒**：设置课程开始前的提醒（如 10 分钟提醒），以应用通知形式推送（贴合公共事件和通知教学内容）；
- **课程表编辑**：支持课程信息的修改、删除，支持课程表的导出（简化为本地文件保存）。

#### 3.4.2 成绩查询

- 支持用户输入学号、密码（模拟登录，无实际后台），查询课程成绩；
- 以列表形式展示课程名称、成绩、绩点、考试时间，支持成绩排序、筛选。

#### 3.4.3 图书馆检索

- 支持图书名称、作者、ISBN 号搜索，展示图书馆藏信息（藏书位置、剩余数量）；
- 支持图书收藏，标记想看 / 已借图书。

> **技术点覆盖**：自定义组件（课程表组件）、本地数据存储（课程表、成绩、收藏图书）、单选框 / 复选框组件使用、事件绑定、通知推送。

### 3.5 生活服务模块

#### 3.5.1 失物招领

- **失物发布**：支持用户发布失物信息（物品名称、丢失地点、丢失时间、物品描述、联系电话），支持上传物品图片（本地图片读取，贴合多媒体开发、本地文件读取教学内容）；
- **拾物发布**：支持用户发布拾物信息，内容与失物发布一致；
- **信息查询**：支持失物 / 拾物信息的分类筛选（如校园卡、书包、电子产品）、关键词搜索，展示信息列表和图片缩略图。

#### 3.5.2 校园美食

- 展示校园食堂、校园周边美食的推荐信息（名称、位置、特色菜品、评分）；
- 支持美食收藏、评分（1-5 星），支持上传美食图片。

#### 3.5.3 音乐播放器（贴合课程大纲开发音乐播放器教学内容）

- 集成简易音乐播放器，支持读取本地音频文件，实现播放、暂停、上一曲、下一曲功能；
- 实现播放器进度条展示、歌曲信息（歌名、歌手）展示，支持播放器悬浮窗播放（贴合 Stage 模型悬浮窗、多媒体开发教学内容）。

> **技术点覆盖**：本地图片 / 音频文件读取、多媒体组件（Image/Video/Audio）开发、悬浮窗功能、自定义评分组件、本地数据存储（失物招领、美食信息）。

### 3.6 我的模块

#### 3.6.1 个人信息管理

- 展示用户个人信息（头像、昵称、学号、学院、专业），支持头像上传（本地图片）、个人信息修改；
- 支持密码修改（模拟登录密码，本地存储）。

#### 3.6.2 我的收藏

- 统一展示收藏的校园公告、资讯、图书、美食等信息，支持收藏信息的删除、分类查看。

#### 3.6.3 待办事项管理

- 支持待办事项的新增（输入待办内容、设置提醒时间）、删除、标记完成，支持待办事项的分类（学习、生活、其他）；
- 支持待办事项的批量操作（批量删除、批量标记完成）。

#### 3.6.4 应用设置

- **通用设置**：支持 APP 主题切换（浅色 / 深色）、字体大小调整（小 / 中 / 大）；
- **通知设置**：支持开启 / 关闭课程提醒、公告提醒、待办提醒；
- **清理缓存**：支持清理 APP 本地缓存的图片、音频、数据文件；
- **关于我们**：展示 APP 版本、开发团队信息、课程相关说明。

> **技术点覆盖**：自定义头像上传组件、本地数据存储与管理、主题切换功能、开关组件（Toggle）使用、批量操作事件实现。

## 四、非功能需求

### 4.1 性能需求

- 应用启动时间 ≤ 3 秒，页面跳转响应时间 ≤ 1 秒，无明显卡顿；
- 支持本地图片、音频文件的快速加载，图片加载无明显失真，音频播放无卡顿、断音；
- 应用在后台运行时，悬浮窗、通知提醒功能正常，无内存泄漏问题。

### 4.2 易用性需求

- 界面设计简洁、直观，符合高校学生的使用习惯，移动端操作便捷（如滑动、点击、下拉刷新）；
- 所有功能操作步骤不超过 3 步，提供清晰的操作提示，无专业技术门槛；
- 支持界面自适应，在 HarmonyOS NEXT 不同屏幕尺寸的移动端设备上，界面布局合理、无内容遮挡。

### 4.3 稳定性需求

- 应用连续运行 24 小时无崩溃、闪退问题；
- 本地文件（图片、音频、数据）读取 / 保存无失败，数据无丢失；
- 页面频繁跳转、组件多次交互后，应用功能正常，无卡顿、卡死现象。

### 4.4 可维护性需求

- **代码规范**：遵循 HarmonyOS NEXT 开发规范，添加清晰的代码注释，变量、函数、组件命名规范；
- **项目结构**：按模块划分代码目录（如 pages、components、utils、data），结构清晰，便于团队协作开发和后续修改；
- **数据管理**：本地数据按功能模块分类存储，数据格式统一（如 JSON 格式）。

## 五、界面设计需求

### 5.1 设计原则

- **简约化**：采用极简设计风格，减少冗余元素，突出核心功能和信息；
- **一致性**：全 APP 的配色、字体、组件样式、操作方式保持一致，如按钮样式、列表样式、弹窗样式统一；
- **移动端适配**：充分考虑移动端竖屏使用场景，避免横向滚动，合理利用屏幕空间，支持单手操作。

### 5.2 配色规范

- **主色调**：采用清新的蓝色系（#1677FF），贴合校园青春、简洁的风格，用于导航栏、按钮、选中状态等；
- **辅助色调**：浅灰色（#F5F7FA）作为背景色，橙色（#FF7D00）作为提醒色（如待办、通知），绿色（#00B42A）作为完成状态色；
- **文字色调**：深黑色（#1D2129）作为主文字色，浅灰色（#86909C）作为辅助文字色，白色作为深色背景文字色。

### 5.3 字体规范

- 采用 HarmonyOS NEXT 系统默认字体，保证跨设备的兼容性；
- 文字大小：标题（30sp）、正文（28sp）、辅助文字（24sp），符合移动端文字显示舒适度；
- 文字行高：标题行高 1.2 倍，正文行高 1.5 倍，保证文字可读性。

### 5.4 组件设计规范

- 所有按钮、输入框、列表、弹窗等组件遵循 ArkUI 原生组件设计规范，适当自定义样式，不做过度个性化修改；
- **组件间距**：页面内元素间距统一为 16px、24px，模块间距为 32px，保证界面整洁；
- **圆角设计**：所有卡片、按钮、图片采用 8px 圆角，贴合移动端主流设计风格，避免尖锐边角。

## 六、数据需求

### 6.1 数据存储方式

基于初学者开发能力和项目周期，全部采用本地数据存储，不涉及后台服务器开发，使用 HarmonyOS NEXT 的本地存储能力（如 Preferences、文件存储），数据格式以 JSON 为主，便于解析和管理。

### 6.2 数据分类存储

| 数据类型 | 存储方式 |
|---------|---------|
| 用户信息数据（头像、昵称、学号、学院、专业、密码等） | Preferences 轻量级存储 |
| 课程表 / 成绩数据 | JSON 文件存储 |
| 收藏数据（公告、资讯、图书、美食等） | JSON 文件存储 |
| 待办事项数据 | Preferences 存储 |
| 失物招领 / 校园美食数据 | JSON 文件 + 本地文件（图片）存储 |
| 音乐播放器数据 | JSON 文件存储 |

### 6.3 测试数据

开发过程中需准备模拟测试数据，如校园公告、课程表、成绩、失物招领信息等，便于功能测试和界面展示。

## 七、开发与验收标准

### 7.1 开发阶段划分（2 个月）

为保证开发进度，将开发过程分为 4 个阶段，贴合课程导学中的项目式教学设计要求：

| 阶段 | 时间 | 内容 |
|------|------|------|
| 第一阶段 | 第 1-2 周 | 项目导入与基础搭建：需求分析、界面原型设计；搭建 DevEco Studio 开发环境，创建项目，基于 Stage 模型完成基础框架搭建；实现底部 Tab 导航、首页基础布局、通用组件初步开发 |
| 第二阶段 | 第 3-4 周 | 项目基础开发：完成五大模块基础页面开发和布局设计；实现基础组件交互、页面路由跳转、简单本地数据存储；完成校园信息、我的模块核心功能开发 |
| 第三阶段 | 第 5-6 周 | 项目提高开发：完成学习助手、生活服务模块核心功能开发，重点实现课程表、失物招领、音乐播放器；实现自定义组件、悬浮窗、本地文件读取、通知推送等进阶技术点；完成所有功能的事件绑定和交互优化 |
| 第四阶段 | 第 7-8 周 | 项目完善与测试优化：全面测试（功能/兼容性/性能），修复 bug；优化界面设计、操作体验；整理项目代码、开发文档，完成项目验收 |

### 7.2 验收标准

- **功能验收**：所有核心功能实现正常，无缺失功能，功能操作符合需求说明书要求；
- **技术验收**：严格覆盖《移动开发技术》课程大纲中的核心技术点，代码规范、项目结构清晰；
- **性能验收**：应用启动、页面跳转、文件加载速度符合性能需求，无崩溃、闪退、卡顿问题；
- **界面验收**：界面设计符合本说明书的设计规范，布局合理、样式统一、移动端适配良好；
- **文档验收**：提交完整的项目开发文档（含代码注释、开发手册、测试报告）。

## 八、附则

- 本需求说明书为项目开发的核心依据，开发过程中如需修改功能，需经团队协商并告知课程指导教师；
- 开发过程中应严格遵循《移动开发技术》课程教学要求，注重技术点的实际应用，而非单纯的功能实现；
- 项目开发完成后，需向课程指导教师进行项目演示，提交项目代码和相关开发文档，完成期末考核验收。
